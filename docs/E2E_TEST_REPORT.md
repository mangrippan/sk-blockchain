### Step 12 — Submit usulan kenaikan pangkat (`POST /usulan`, `jabatan_tujuan_id=2`)
**Duration**: 67ms · **Status**: ❌ **FAIL**  
**Description**: Dosen mengajukan usulan kenaikan pangkat ke jabatan "Asisten Ahli" (`min_kum: 100`); endpoint ini secara otomatis memilih kegiatan terverifikasi yang belum dipakai, memvalidasi via fungsi SQL `sk.validate_usulan_jabatan`, menghitung `total_poin`, dan men-generate `snapshot_hash`  
**Expected**: `201 Created`, usulan baru berstatus `pending`, `total_poin >= 100`  
**Actual**:
```
HTTP 500 — {"error":"Failed to create usulan","message":"txResult is not defined"}
```
**Root cause** — `backend/routes/v1/usulan.js:883`, di dalam audit-log JSON pada handler `POST /` (pembuatan usulan):
```js
// Line 870-871: comment menjelaskan blockchain recording SUDAH DIHAPUS dari handler ini
// Blockchain recording removed - only record when SK is issued
// Usulan at 'pending' status is not final yet

// Audit log
await client.query(
  `INSERT INTO sk.audit_logs (user_id, action, table_name, record_id, new_values)
   VALUES ($1, $2, $3, $4, $5::jsonb)`,
  [req.user.id, 'CREATE', 'usulan_kenaikan_pangkat', usulan.id, JSON.stringify({
    jabatan_tujuan: targetJabatan.nama,
    jabatan_tujuan_id: jabatan_tujuan_id,
    total_poin: totalPoin,
    kegiatan_count: kegiatanList.length,
    snapshot_hash: snapshotHash,
    blockchain_tx: txResult ? 'success' : 'skipped',   // ← line 883: txResult TIDAK PERNAH dideklarasikan di handler ini
  })]
);
```
Variabel `txResult` adalah sisa referensi dari kode blockchain-recording yang **sudah dihapus** dari handler ini (sesuai komentar baris 870–871 — perekaman blockchain untuk usulan dipindah ke endpoint `terbitkan-sk`, lihat baris 1190 di mana `txResult` *memang* dideklarasikan dengan benar). Tapi referensi ke `txResult` pada baris 883 tertinggal, sehingga **setiap** pemanggilan `POST /usulan` melempar `ReferenceError: txResult is not defined` di dalam blok transaksi SQL, membatalkan `INSERT` dan mengembalikan `HTTP 500`.

**Dampak**: Ini bukan kegagalan blockchain yang ter-toleransi (graceful fallback) — ini adalah **crash JavaScript murni** yang terjadi *sebelum* baris `COMMIT`, sehingga **tidak ada satupun usulan kenaikan pangkat baru yang bisa diajukan** oleh dosen manapun selama bug ini ada di kode. Fitur inti aplikasi (alur "Ajukan Usulan Kenaikan Pangkat") sepenuhnya tidak berfungsi.

**Verifikasi bahwa ini bug pada kode yang sudah di-commit (bukan WIP lokal)**:
```

```
`backend/routes/v1/usulan.js` **tidak** termasuk dalam file yang dimodifikasi pada working tree saat ini (lihat `git status` — hanya `kegiatan.js`, `server.js`, `fabricClient.js`, dll. yang berstatus `M`). ## 🎯 Key Findings

### 1. 🔴 Blockchain writes are silently failing while the status indicator reports "healthy" 🔴

**Severity**: High — affects data integrity guarantees and operator trust in the new monitoring feature

**Bukti langsung dari run ini**:
- 3 dari 3 kegiatan yang diverifikasi (Steps 8–10) berhasil ter-update di database (`status=verified`) tapi **tidak satupun** mendapat `tx_id_fabric` — dikonfirmasi baik dari respons API maupun query langsung ke `sk.kegiatan_dosen`
- Pada saat yang hampir bersamaan, `GET /system/status` melaporkan `blockchain.connected: true` dengan `identity: "appUser"`, `channel: "primachannel"` — tampak benar-benar sehat

**Root cause (ditelusuri via skrip diagnostik sementara `diag-fabric.js`, dijalankan langsung memanggil `fabricClient.recordKegiatanCreation`, lalu dihapus setelah selesai)**:
```
Failed to connect before the deadline on Endorser- name: peer0.org1.example.com, url:grpcs://localhost:7051
Broadcast Client peer0.org1.example.com grpcs://localhost:7051 is not connected
❌ Fabric submitTransaction(CreateKegiatan) failed: No valid responses from any peers
```

**Bukti timeline (cross-check `connectedAt` API ↔ histori `verified_at`/`tx_id_fabric` di DB ↔ `docker inspect` container Fabric)** — diurutkan secara kronologis:

| Waktu (UTC) | Peristiwa | Sumber |
|---|---|---|
| 2026-05-20 ~04:02–12:40 | 6 kegiatan diverifikasi **dengan** `tx_id_fabric` terisi (sesi gateway *sebelumnya*, sudah lama digantikan) | Query `sk.kegiatan_dosen` |
| 2026-06-06 **10:33:12** | Backend connect ke Fabric gateway sesi yang **masih aktif sekarang** (`identity: appUser`) — `connectedAt` dari `/system/status`; selaras dengan `uptime` backend (~73,138 detik mundur dari `runAt` 06:51:54 UTC → proses backend mulai ≈ 10:32:56 UTC — hampir bersamaan dengan `connectedAt`, sesuai pola "connect ke Fabric saat startup") | `GET /system/status`, `GET /health` |
| 2026-06-06 **11:12:29** | Kegiatan `89b42f0c-...` diverifikasi — **gagal** tulis on-chain (`tx_id_fabric IS NULL`); ini adalah verifikasi **pertama** yang gagal dalam sesi gateway saat ini, hanya **~39 menit** setelah `connectedAt` | Query `sk.kegiatan_dosen` |
| 2026-06-06 **11:25:48–11:25:49** | Container `orderer.example.com` dan `peer0.org1.example.com` menunjukkan `StartedAt` (waktu mulai instance container yang **sedang berjalan saat ini**) — ≈13 menit **setelah** kegagalan pertama tercatat | `docker inspect --format='{{.State.StartedAt}}'` |

**Kesimpulan**: Gateway sesi saat ini (terhubung `10:33:12`, masih dilaporkan `connected: true` saat ini — lebih dari 20 jam kemudian) **berhenti mampu menyelesaikan transaksi hanya ~39 menit setelah terhubung**, dan **tidak pernah pulih** sejak itu — terbukti dari nol keberhasilan tulis on-chain dalam rentang hampir 20 jam yang mencakup ketiga verifikasi pada run E2E ini. `docker inspect` menunjukkan jaringan Fabric (peer & orderer) telah di-*restart* setidaknya sekali (`StartedAt ≈ 11:25:48`) dalam rentang waktu tersebut — selaras dengan, meski tidak secara definitif membuktikan tunggal sebagai penyebab, hipotesis bahwa `gateway`/`contract` yang di-cache backend menjadi *stale* relatif terhadap jaringan Fabric yang berjalan saat ini. ### Root Cause #1 — Sertifikat TLS CA basi pasca-regenerasi jaringan Fabric

Jaringan `test-network` Fabric ternyata **diregenerasi total** pada `~2026-06-06 11:21 UTC` (membuat root CA Org1 yang baru — sertifikat baru bertanda `Not Before: Jun 6 11:21:00 2026`), tetapi `fabric-config/connection-org1.json` dan `connection-org1-wsl.json` (dipakai backend untuk membangun koneksi gRPC ke peer & CA) masih berisi **sertifikat TLS CA dari root lama** (`Not Before: May 30 09:41:00 2026`) — root yang sudah tidak ada lagi di trust-store MSP peer manapun di jaringan yang berjalan sekarang.

Kedua root memiliki **CN/DN yang identik** (`CN=ca.org1.example.com`) sehingga tidak langsung kentara dari namanya saja — tetapi merupakan **instance kriptografis yang sama sekali berbeda** (serial number, fingerprint, dan masa berlaku berbeda total). Setiap *handshake* TLS yang mencoba memverifikasi sertifikat peer terhadap root basi ini ditolak di level transport, dan SDK `fabric-network` melaporkannya sebagai `Failed to connect before the deadline on Endorser...` / `Broadcast Client ... is not connected` — persis pola error yang tercatat pada Finding #1 di atas.

**Perbaikan**:
- Mengganti blok `tlsCACerts.pem` untuk `peer0.org1.example.com` dan `ca.org1.example.com` di kedua connection profile (`connection-org1.json` dan `connection-org1-wsl.json`) dengan sertifikat root CA yang sedang berlaku — byte-identik dengan profil auto-generated milik network saat ini (`fabric-network/fabric-samples/test-network/organizations/peerOrganizations/org1.example.com/connection-org1.json`)
- Memperkuat `connectGateway()` di `backend/utils/fabricClient.js` agar **tidak lagi menelan kegagalan koneksi secara diam-diam**: ternyata `gateway.connect()` *tidak* melempar exception meski endpoint gRPC gagal terhubung (ia "berhasil" dengan endpoint yang rusak di baliknya). Kode sekarang secara eksplisit memeriksa `channel.getEndorsers()`/`getCommitters()` setelah connect dan melempar error bila ada endpoint yang `connected !== true` — sehingga `getConnectionStatus()` (Fix B / Finding #1) tidak bisa lagi melaporkan `connected: true` palsu berdasarkan objek yang di-cache semata

**Verifikasi**: setelah penggantian sertifikat dan restart backend, `gateway.connect()` berhasil membangun koneksi TLS yang valid ke seluruh endpoint (peer, orderer, CA), dan baik `evaluateTransaction` maupun `submitTransaction` langsung berhasil pada percobaan pertama — tidak ada lagi error handshake.

### Root Cause #2 — `AjukanUsulanKenaikanPangkat` / `ProsesUsulanKenaikanPangkat` tidak pernah dipanggil dari route (lifecycle wiring gap)

Setelah konektivitas TLS pulih, percobaan menerbitkan SK (`TerbitkanSkKenaikanPangkat`) tetap gagal — kali ini dengan pesan dari chaincode sendiri: **"Usulan {id} does not exist"**. Penelusuran `backend/routes/v1/usulan.js` menunjukkan handler `POST /` (pengajuan usulan) menyisakan komentar `// Blockchain recording removed...` — `fabricClient.recordUsulanCreation` **tidak pernah dipanggil saat usulan diajukan**, sehingga chaincode tidak pernah punya record `pending` untuk usulan tsb sama sekali. Lifecycle on-chain yang diasumsikan chaincode (`AjukanUsulanKenaikanPangkat → ProsesUsulanKenaikanPangkat → TerbitkanSkKenaikanPangkat`, dengan validasi `status` di setiap transisi) tidak pernah benar-benar dimulai.

**Perbaikan #1 (dipilih user — opsi *"Wire up recordUsulanCreation saat submit"*)**: menambahkan pemanggilan `fabricClient.recordUsulanCreation(...)` di handler `POST /usulan` (sebelum insert audit log), dibungkus `try/catch` + cek `fabricClient.isFabricEnabled()`, mengikuti pola *graceful fallback* yang sudah konsisten dipakai di endpoint lain — kegagalan blockchain tidak menggagalkan operasi DB, dan `tx_id_fabric` hanya di-update bila transaksi benar-benar berhasil.

Setelah perbaikan ini diverifikasi (usulan kini benar-benar ada on-chain dengan `status: pending`), percobaan `TerbitkanSkKenaikanPangkat` berikutnya **maju ke lapisan masalah berikutnya** — gagal dengan pesan baru: *"Usulan {id} must be in 'diproses' status to issue SK (current: pending)"*. Penelusuran menemukan **gap yang persis sama, satu langkah lebih jauh di lifecycle**: handler `PUT /usulan/:id/proses` juga tidak pernah memanggil `fabricClient.recordUsulanProcess` (transisi on-chain `pending → diproses`).

**Perbaikan #2 (dipilih user — opsi *"Ya, wire up recordUsulanProcess juga di route /proses"*)**: menambahkan `fabricClient.recordUsulanProcess(req.params.id, req.user.id)` ke handler `PUT /usulan/:id/proses`, mengikuti pola identik (try/catch, guard `isFabricEnabled()`, update kondisional `tx_id_fabric`, field `blockchain_tx` di audit log).

**Verifikasi**: query langsung ke chain (`fabricClient.getUsulan(id)`, melewati lapisan DB sepenuhnya) mengonfirmasi rangkaian transisi status on-chain `pending → diproses → sk_issued` lengkap dan benar untuk usulan uji.

### Root Cause #3 — Identitas `appUser` tidak memiliki atribut ABAC `role` (akar masalah arsitektural terdalam)

Setelah Root Cause #2 diperbaiki, `TerbitkanSkKenaikanPangkat` (dan ternyata juga `ProsesUsulanKenaikanPangkat`, `TolakUsulanKenaikanPangkat`, `VerifyKegiatan`, `GetAllKegiatan`) tetap gagal — kali ini dengan: **`Authorization failed: No role attribute found in client identity certificate`**.

Penelusuran ke `chaincode/lib/kegiatanContract.js` (helper `_checkRole`, baris 24–53) menunjukkan fungsi-fungsi tsb membaca atribut kustom `role` dari sertifikat X.509 pemanggil via `ctx.clientIdentity.getAttributeValue('role')` — sebuah mekanisme **ABAC (Attribute-Based Access Control)** yang sama sekali berbeda dari konsep "role" di level registrasi CA (`ca.register({role: 'client'})`, yang hanya menentukan *tipe identitas* CA, bukan klaim atribut yang tertanam di sertifikat).

`fabric-network/enrollUser.js` ternyata mendaftarkan `appUser` **tanpa** klaim atribut ini sama sekali — sehingga sertifikat yang diterbitkan untuk identitas tunggal yang dipakai backend untuk *semua* transaksi (pola "single shared service identity": otorisasi per pengguna akhir sepenuhnya ditangani di middleware Express `checkRole()`, bukan on-chain) tidak pernah lolos `_checkRole()` chaincode.

**Perbaikan (dipilih user — opsi *"Tambah attrs role di enrollUser.js"*, dengan nilai atribut `admin_sdm` dikonfirmasi eksplisit oleh user di antara dua opsi yang diajukan: `admin_sdm` vs `superadmin`)**:
- `enrollUser.js` (perbaikan permanen untuk re-enrollment di masa depan): menambahkan `attrs: [{name:'role', value:'admin_sdm', ecert:true}]` ke `ca.register()`, `attr_reqs: [{name:'role', optional:false}]` ke `ca.enroll()`, dan perbaikan tambahan untuk URL CA (rewrite `localhost`→`127.0.0.1`, mengatasi `ETIMEDOUT ::1:7054` — Docker Desktop di Windows tidak konsisten meneruskan port CA lewat IPv6 loopback)
- Untuk identitas `appUser` yang **sudah terlanjur** terdaftar tanpa atribut ini — dan tidak bisa dihapus untuk didaftar ulang karena server CA dikonfigurasi `Identity removal is disabled` (code 56) — digunakan `IdentityService.update()` untuk menambahkan atribut + secret baru pada registrasi yang sudah ada (setelah menaikkan `maxEnrollments` ke `-1`, karena batas enrollment lama yaitu `1` sudah habis terpakai dan menyebabkan `Authentication failure` generik pada percobaan re-enroll pertama), lalu `ca.enroll()` ulang dengan `attr_reqs` untuk benar-benar menerbitkan sertifikat baru yang membawa atribut tsb. Tindakan menghapus/mendaftar-ulang identitas ini bersifat destruktif terhadap state CA — izin eksplisit diminta ke user terlebih dahulu sebelum dijalankan

**Verifikasi**: `openssl x509` pada sertifikat baru menunjukkan extension `1.2.3.4.5.6.7.8.1: {"attrs":{"role":"admin_sdm"}}` tertanam dengan benar, dan ditandatangani oleh root CA yang sedang berlaku saat ini (`Not Before: Jun 6 11:21:00 2026 GMT` — cocok dengan root CA baru dari Root Cause #1).

### Catatan diagnostik — jebakan "PASS palsu" akibat pola overwrite kondisional pada `tx_id_fabric`

Selagi memverifikasi perbaikan di atas, satu run E2E sempat melaporkan langkah "Check SK transaction recorded on blockchain" sebagai **PASS** — padahal query langsung ke chain (`fabricClient.getUsulan`) menunjukkan record on-chain **masih `status: pending`, `skHash: null`**. `tx_id_fabric` yang tampil di DB ternyata adalah **sisa dari transaksi pembuatan usulan** (Root Cause #2), bukan transaksi penerbitan SK — `recordUsulanSKIssued` sebenarnya gagal (mengembalikan `null`), tapi pola `if (txResult) { ...update tx_id_fabric... }` di `backend/routes/v1/usulan.js` membuat kegagalan ini **tidak pernah menimpa nilai lama**, sehingga kolom tetap berisi tx_id valid dari transaksi sebelumnya — secara visual tidak bisa dibedakan dari keberhasilan asli.

Pola ini **tidak diubah** dalam sesi ini (di luar scope perbaikan yang diminta user — dan setelah Root Cause #1–#3 diperbaiki, transaksi penerbitan SK memang berhasil sehingga nilai `tx_id_fabric` yang akhirnya tersimpan benar adanya). Dicatat di sini sebagai **jebakan diagnostik untuk investigasi berikutnya**: pola "update kolom hanya bila hasil sukses" dapat menyembunyikan kegagalan di balik nilai lama yang tampak sah, dan satu-satunya cara mendiagnosisnya secara pasti adalah query langsung ke chain — melewati DB sepenuhnya.

### Status akhir

**Tujuan investigasi ini — "transaksi blockchain sungguhan harus berhasil lagi" — tercapai dan terverifikasi penuh, di banyak lapisan independen.** Keempat fungsi lifecycle chaincode yang benar-benar terhubung ke route (`AjukanUsulanKenaikanPangkat`/create, `ProsesUsulanKenaikanPangkat`/process, `TerbitkanSkKenaikanPangkat`/SK-issue, `CreateKegiatan`/kegiatan) kini berhasil dieksekusi sebagai transaksi chain sungguhan, end-to-end, terhadap jaringan live.

Dengan ini:
- **Finding #1 dan #2 pada laporan asli di atas dinyatakan RESOLVED** (sudah diperbaiki sebelum sesi addendum ini dimulai — lihat commit history; addendum ini berfokus pada root cause *konektivitas* yang baru terungkap setelah keduanya diperbaiki)
- **Verdict laporan berubah dari ❌ NOT PRODUCTION READY menjadi ✅ siap untuk lapisan blockchain-nya** — dengan catatan rekomendasi desain di Recommendations §3 (liveness check sungguhan untuk `getConnectionStatus`, job rekonsiliasi DB↔chain) tetap relevan sebagai peningkatan operasional jangka panjang, bukan blocker rilis
- `TolakUsulanKenaikanPangkat`/`recordUsulanRejection` dan `VerifyKegiatan`/`recordKegiatanVerification` tetap belum terhubung ke route manapun — ini konsisten dengan desain yang sudah ada sebelumnya (bukan regresi atau temuan baru), dan berada di luar scope investigasi ini
